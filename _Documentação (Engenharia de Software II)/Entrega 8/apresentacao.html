Contexto &
Objetivo

Modernizando a gestão escolar através da centralização de dados.

Meta Principal

Otimizar matrículas, diários eletrônicos e relatórios gerenciais em uma interface única.

O Problema

  • • Descentralização crítica de dados.
  • • Excesso de processos manuais e uso de papel.
  • • Dificuldade na geração de relatórios confiáveis.

A Solução (GUTO)

  • • Plataforma Web Centralizada.
  • • Gestão unificada (Acadêmica + Administrativa).
  • • Acesso para Secretários, Professores e Direção.

Arquitetura do Sistema

Estilo Monólito Modular (Django MVT)

Optamos pela arquitetura Model-View-Template do Django para garantir alta produtividade, coesão com o domínio escolar complexo e facilidade de deploy.

  • Modularização por Apps
  • Separação de Responsabilidades
  • Escalabilidade Vertical
1. Apresentação (Templates/HTMX)
2. Aplicação (Views/Forms)
3. Domínio (Models/Services)
4. Dados (SQLite/PostgreSQL)

Padrões de Persistência e Design

Decisões técnicas fundamentais de Engenharia de Software

Active Record

Padrão nativo do Django ORM. O objeto carrega dados e comportamento de persistência.

aluno = Aluno(nome="Guto")
aluno.save()

Utilizado em: CRUD Básico

Repository Pattern

Camada intermediária para isolar a lógica de acesso a dados de consultas complexas.

repo = AlunoRepository()
lista = repo.get_ativos_por_turma(id)

Implementado em: repository_layer.py

Singleton

Garante uma instância única para conexões de banco de dados e configurações globais do sistema.

Factory

Utilizado para instanciar os repositórios de forma desacoplada, facilitando a injeção de dependências.
